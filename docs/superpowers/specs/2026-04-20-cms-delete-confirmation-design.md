# CMS Delete Confirmation — Design Spec

**Date:** 2026-04-20  
**Status:** Approved

---

## Problem

All four CMS delete actions currently use `window.confirm()` — a native browser dialog that is visually inconsistent with the app's design, cannot be styled, and blocks the main thread.

Locations:

- `pages/cms/pages/index.tsx` — delete static page
- `pages/cms/posts/index.tsx` — delete article/post
- `pages/cms/galleries/index.tsx` — delete gallery
- `pages/cms/galleries/edit.tsx` — delete individual image from a gallery

---

## Goal

Replace every `window.confirm()` call with a styled `AlertDialog` that:

1. Matches the app's design system (shadcn/ui)
2. Cannot be dismissed by clicking outside (appropriate for destructive actions)
3. Shows the item name in the dialog title for context
4. Is implemented as a single reusable component used across all four locations

---

## Approach

Use shadcn's `AlertDialog` primitive (semantically correct for irreversible actions), installed via the shadcn CLI. Wrap it in a reusable `ConfirmDeleteDialog` component.

---

## Architecture

### New files

**`resources/js/components/ui/alert-dialog.tsx`** (generated by shadcn)  
The raw AlertDialog primitives. Treated as vendored UI — do not edit manually.

**`resources/js/components/confirm-delete-dialog.tsx`** (new)  
A thin wrapper with the app's text and styling conventions.

```tsx
interface Props {
    open: boolean;
    onOpenChange: (open: boolean) => void;
    title: string;
    description?: string; // defaults to "Tindakan ini tidak bisa dibatalkan."
    onConfirm: () => void;
    processing?: boolean;
}
```

Usage example:

```tsx
<ConfirmDeleteDialog
    open={confirmOpen}
    onOpenChange={setConfirmOpen}
    title={`Hapus halaman "${selectedPage?.title}"?`}
    onConfirm={handleDelete}
/>
```

### State pattern per page

Each index page adds two state variables:

```tsx
const [confirmOpen, setConfirmOpen] = useState(false);
const [pendingItem, setPendingItem] = useState<PageType | null>(null);
```

The existing `handleDelete(item)` function changes from:

```tsx
// Before
function handleDelete(page: Page) {
    if (!window.confirm(`Hapus halaman "${page.title}"?`)) return;
    router.delete(PageController.destroy.url(...));
}
```

To a two-step flow:

```tsx
// After — step 1: open dialog
function handleDelete(page: Page) {
    setPendingItem(page);
    setConfirmOpen(true);
}

// After — step 2: execute on confirm
function executeDelete() {
    if (!pendingItem) return;
    router.delete(
        PageController.destroy.url({
            current_team: teamSlug,
            page: pendingItem.id,
        }),
        { preserveScroll: true, onFinish: () => setConfirmOpen(false) },
    );
}
```

The `ConfirmDeleteDialog` is rendered once at the bottom of each component's JSX.

---

## AlertDialog behavior

- **Cannot be dismissed by clicking outside** — intentional for destructive actions.
- **Escape key** closes the dialog (shadcn default, acceptable).
- The "Hapus" button is `variant="destructive"`.
- The "Batal" button closes without action.
